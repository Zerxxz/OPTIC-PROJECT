Multi-agent system

The three specialists

Each agent is a pure function. They negotiate, the risk agent has veto power, and the executor does defence-in-depth validation before any PTB is signed.

quant.sui

Quant

Reads the DeepBook orderbook + indexer, evaluates a pluggable strategy (mean-reversion, momentum, market-making, or custom), and proposes a trade with a confidence score.

  • Inputs: Agent state, market state, strategy blob
  • Output: place_order or no_op
  • Confidence: 0–10,000 bps
  • Strategies: mean-reversion, momentum, market-making
risk.sui

Risk

Monitors realized vol, drawdown, and exposure. Has veto power: can pause the agent entirely or open a DeepBook Predict hedge to cap downside.

  • Circuit breaker: -$50 daily loss → pause
  • High vol: > 8% → open NO hedge (1% of treasury)
  • Strike: 5% OTM put, 24h expiry
  • Veto: if paused, every other action is suppressed
executor.sui

Executor

Defence-in-depth gate. Re-checks every risk param, sizes within exposure limits, and composes the final PTB. Logs an AuditEntry to Walrus.

  • Defence in depth — never trust upstream
  • Idempotent dispatch
  • Audit log on every action
  • PTB composition: tx.moveCall chained

Merge & validate

The orchestrator combines specialist outputs deterministically.

Risk has veto

If risk.decide() returns pause, the orchestrator halts the agent regardless of what quant proposed. The pause action is dispatched and an audit entry is logged.

Highest confidence wins

When both quant and risk return non-pause actions, the decision with the higher confidenceBps is selected. The runner-up is logged but not executed.

Validate (defence in depth)

Before dispatch, the orchestrator re-validates: size ≤ max position, price ≠ 0, treasury ≥ hedge size, status = active. A failure here is logged as decision_rejected.

Idempotent PTB

The PTB is deterministic for the same state + market. Re-running the same cycle produces the same on-chain call sequence. This is critical for audit replay.
